# Plan: Fix Routing & Prepare GitHub Pages Deployment

## Root Cause Analysis

The current setup uses `BrowserRouter` with `basename="/ePortfolio"` plus a SPA fallback mechanism (`public/404.html` + `index.html` restoration script) to support deep linking on GitHub Pages. Two issues cause instability:

1. **No trailing slash normalization**: GitHub Pages may serve `/ePortfolio` (no slash) or `/ePortfolio/` (with slash) depending on the request. The `index.html` restoration script at line 27 does `l.pathname.slice(0, -1)` which assumes a trailing slash — it would incorrectly slice the last character if the URL ends without one (e.g., `/ePortfolio` → `/ePortfoli`).

2. **Fragile basename passthrough**: `App.tsx` line 44 calls `sectionIndexByPath(location.pathname, '')` with an empty basename string. This works only because React Router strips the basename from `location.pathname`, but it's a latent fragility.

## Recommended Strategy: HashRouter

For a static student portfolio hosted on GitHub Pages (no server-side routing control), **HashRouter** is the cleanest and most reliable solution:

- Eliminates all SPA fallback complexity (`404.html`, `index.html` script)
- Zero trailing slash concerns (hash is appended, never sent to server)
- No `basename` prop needed
- All existing `<NavLink>`, `<Link>`, `<Route>` work identically
- `Vite base` stays for asset resolution (JS, CSS, fonts)
- URLs: `/ePortfolio/#/introduction`, `/ePortfolio/#/engagement`, etc.
